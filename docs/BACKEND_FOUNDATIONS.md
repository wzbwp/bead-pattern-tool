# 后端工程基础：从能调通到能稳定调通

这份文档面向已经会写前端、会调用接口，但想系统补齐后端工程能力的人。目标不是罗列名词，而是建立一条清晰主线：

```txt
一次请求如何稳定完成
  -> 多个请求如何并发完成
  -> 状态如何可靠保存和读取
  -> 多个服务如何协作完成
  -> 出问题时如何看见、定位、恢复
```

尤其是在 AI 应用、多 Agent 协作、记忆系统、任务编排里，这些能力不是加分项，而是地基。

## 总览

| 模块 | 解决的问题 | 如果缺失会怎样 |
| --- | --- | --- |
| 网络基础 | 服务之间如何可靠通信 | 本地能跑，线上偶发失败、重复提交、鉴权混乱 |
| 并发编程 | 如何同时处理大量任务 | 阻塞、雪崩、连接耗尽、请求排队不可控 |
| 存储和缓存 | 状态、记忆、索引如何管理 | 数据不一致、缓存脏读、查询慢、记忆不可追踪 |
| 分布式系统 | 多服务、多任务如何协作 | 消息丢失、重复消费、状态分裂、故障扩散 |
| 可观测性 | 如何知道系统发生了什么 | 线上问题只能猜，修复靠运气 |

## 1. 网络基础

网络基础是一切后端工程的入口。前端里“调接口”通常只看到 `fetch` 或 `axios`，但后端关注的是：请求如何被认证、如何超时、失败后是否重试、重试会不会重复执行、链路中每一跳是否可控。

### 1.1 HTTP

HTTP 是最常见的服务通信协议。

核心概念：

- Method：`GET`、`POST`、`PUT`、`PATCH`、`DELETE`
- Status Code：`2xx` 成功，`4xx` 客户端错误，`5xx` 服务端错误
- Header：认证、内容类型、缓存、追踪 ID 等元数据
- Body：请求或响应数据
- Connection：短连接、长连接、HTTP/2 多路复用

常见语义：

- `GET`：读取资源，应尽量无副作用
- `POST`：创建资源或触发动作，通常不是天然幂等
- `PUT`：整体替换资源，通常可以设计成幂等
- `PATCH`：局部修改资源
- `DELETE`：删除资源，可以设计成幂等

工程要点：

- 明确接口的资源语义，不要所有操作都写成 `POST /doSomething`
- 对错误码建立约定，例如参数错误用 `400`，未登录用 `401`，无权限用 `403`
- 每个请求生成 `request_id`，贯穿日志、调用链、错误响应
- 请求体要限制大小，避免大文件或恶意请求拖垮服务
- 对外接口要考虑版本，例如 `/api/v1/tasks`

常见坑：

- 把业务错误都返回 `200`
- 服务端没有超时控制，导致请求无限挂住
- 客户端重试 `POST`，结果重复创建订单或任务
- 没有统一错误结构，前端只能靠字符串判断

推荐错误响应结构：

```json
{
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "Task not found",
    "request_id": "req_01H..."
  }
}
```

### 1.2 RPC

RPC 是 Remote Procedure Call，远程过程调用。HTTP 偏资源，RPC 偏函数调用。

常见形式：

- REST over HTTP
- JSON-RPC
- gRPC
- GraphQL
- 内部自定义协议

RPC 的优势：

- 适合服务内部高频通信
- 接口契约更明确
- gRPC 这类协议支持强类型、代码生成、流式通信

RPC 的风险：

- 容易让远程调用看起来像本地函数，忽略网络失败
- 链路变长后，超时和重试会变得复杂
- 强依赖接口契约，版本升级需要更谨慎

工程原则：

- 永远把 RPC 当成不可靠调用
- 每个 RPC 都必须有超时
- 调用方要区分“可重试错误”和“不可重试错误”
- 服务端接口要考虑幂等
- 内部接口也要有鉴权或服务身份认证

### 1.3 认证与授权

认证回答“你是谁”，授权回答“你能做什么”。

常见认证方式：

- Session Cookie
- JWT
- OAuth 2.0
- API Key
- mTLS
- 内部服务 Token

Session Cookie：

- 服务端保存会话状态
- 浏览器友好
- 易于主动失效
- 需要考虑 CSRF

JWT：

- Token 自包含用户信息和过期时间
- 适合分布式服务校验
- 主动失效较麻烦
- 不要把敏感信息放进 Payload

API Key：

- 常用于开放平台、机器调用
- 要支持轮换、撤销、限流
- 不应直接等同于用户密码

授权模型：

- RBAC：基于角色，例如 admin、editor、viewer
- ABAC：基于属性，例如资源所有者、部门、时间、环境
- ACL：对单个资源配置访问列表

工程要点：

- 认证失败返回 `401`
- 权限不足返回 `403`
- Token 要有过期时间
- 高风险操作要做二次确认或更强校验
- 后端必须做权限判断，不能只依赖前端隐藏按钮

### 1.4 超时

超时是稳定性的第一道防线。没有超时的系统，失败会变成阻塞，阻塞会耗尽线程、连接和内存。

常见超时：

- 连接超时：建立连接花太久
- 读超时：服务端迟迟不返回数据
- 写超时：请求体发送太久
- 总超时：整个请求最大耗时
- 队列超时：排队等待处理的最大时间

实践建议：

- 所有外部调用必须设置超时
- 上游超时要大于下游超时，留出错误处理时间
- 不同接口使用不同超时，不要全局一个超大值
- 慢任务不要占用同步 HTTP 请求，改为异步任务

示例：

```txt
用户请求总超时：3s
  -> API 服务调用检索服务：1s
  -> API 服务调用模型服务：2s
  -> API 服务保留 300ms 做降级响应
```

### 1.5 重试

重试可以提升成功率，也可能放大故障。

适合重试的情况：

- 网络抖动
- 连接重置
- 临时 `502`、`503`、`504`
- 下游返回明确可重试错误

不适合盲目重试的情况：

- 参数错误
- 权限错误
- 余额不足
- 非幂等写操作
- 下游已经过载

重试策略：

- 固定间隔：简单但容易形成流量尖峰
- 指数退避：每次等待更久
- 抖动 jitter：随机化等待时间，避免同时重试
- 最大次数：防止无限重试

推荐策略：

```txt
最多重试 2-3 次
使用指数退避
加入随机抖动
只重试明确可重试错误
```

### 1.6 幂等

幂等是指同一个操作执行一次和执行多次，最终效果一致。

为什么重要：

- 客户端可能重复点击
- 网络超时后客户端可能重试
- 消息队列可能重复投递
- 服务重启后任务可能重新执行

常见做法：

- 幂等键：`Idempotency-Key`
- 唯一约束：数据库唯一索引
- 状态机：只允许合法状态流转
- 去重表：记录已处理请求或消息
- 业务单号：订单号、任务号、事件号天然去重

例子：

```txt
POST /orders
Idempotency-Key: user_123_create_order_abc
```

服务端处理：

1. 收到请求后查询幂等键是否存在。
2. 如果已存在，直接返回上次结果。
3. 如果不存在，创建订单并保存幂等记录。
4. 幂等记录和订单创建要在同一事务里完成。

常见坑：

- 只在内存里去重，服务重启后失效
- 幂等记录和业务写入不是原子操作
- 幂等键粒度太粗，误伤正常请求
- 幂等键永不过期，存储无限增长

## 2. 并发编程

前端的 `async/await` 只是并发编程的入口。后端面对的是大量请求、连接、任务、队列、文件、数据库操作同时发生，需要理解异步模型、连接池、事件循环和背压。

### 2.1 并发与并行

并发：

- 多个任务在同一时间段内交替推进
- 重点是结构和调度
- 单核也可以并发

并行：

- 多个任务在同一时刻真正同时执行
- 需要多核或多机器

例子：

```txt
一个服务同时处理 1000 个 HTTP 请求：并发
4 个 CPU 核同时做图像处理：并行
```

### 2.2 异步

异步适合 I/O 密集任务：

- HTTP 请求
- 数据库查询
- Redis 访问
- 文件读写
- 消息队列

异步不等于更快。它的价值是等待 I/O 时不阻塞执行线程。

常见问题：

- 忘记 `await`，导致错误丢失
- 并发无限制，瞬间打爆数据库
- 异步任务没有取消机制
- 后台任务失败无人知道

实践建议：

- 给并发数量设置上限
- 所有异步任务都要有错误处理
- 长任务支持取消或过期
- 不要在请求结束后悄悄跑关键业务逻辑，除非有任务系统兜底

### 2.3 连接池

连接池用于复用昂贵连接。

常见连接池：

- 数据库连接池
- Redis 连接池
- HTTP client 连接池
- gRPC channel 池

为什么需要连接池：

- 建连成本高
- 数据库连接数有限
- 复用连接可以降低延迟

连接池关键参数：

- 最大连接数
- 最小空闲连接数
- 获取连接超时
- 空闲连接回收时间
- 连接最大生命周期

常见坑：

- 连接池太小，请求大量排队
- 连接池太大，数据库被连接压垮
- 没有释放连接，连接泄漏
- 慢查询长期占用连接

经验原则：

```txt
连接池大小不是越大越好。
它应该由数据库承载能力、服务实例数、请求耗时共同决定。
```

### 2.4 事件循环

事件循环是 Node.js、浏览器、Python asyncio 等运行时的核心模型。

事件循环适合：

- 大量 I/O
- 短小非阻塞任务
- 网络服务

不适合：

- 长时间 CPU 计算
- 大型同步循环
- 阻塞式文件操作

在事件循环中，最危险的是阻塞主线程。

常见阻塞：

- 大 JSON 序列化/反序列化
- 同步压缩
- 大量正则计算
- 图像处理
- 加密计算

解决方式：

- 拆分任务
- 使用 Worker
- 放到任务队列
- 使用专门计算服务
- 限制输入大小

### 2.5 背压

背压是指下游处理不过来时，上游要放慢、排队、丢弃或降级。

没有背压会发生：

- 请求堆积
- 内存增长
- 队列无限膨胀
- 延迟越来越高
- 最终雪崩

常见背压手段：

- 限流
- 队列长度限制
- 并发数限制
- 超时丢弃
- 降级响应
- 熔断
- 批处理

例子：

```txt
上传 10000 张图片需要生成向量。
不能直接同时调用模型服务。
应该进入任务队列，由 worker 按固定并发消费。
队列过长时拒绝新任务或提示稍后再试。
```

### 2.6 并发安全

并发下容易出问题的地方：

- 余额扣减
- 库存扣减
- 任务状态更新
- 缓存刷新
- 文件写入
- 幂等记录

常见保护方式：

- 数据库事务
- 行锁
- 乐观锁
- 唯一索引
- 分布式锁
- 状态机约束

不要优先迷信分布式锁。很多问题用数据库唯一约束和事务更简单可靠。

## 3. 存储和缓存

存储解决“事实在哪里”，缓存解决“如何更快访问”。AI 应用里的记忆系统、任务状态、用户上下文、向量索引，本质上都依赖存储设计。

### 3.1 关系型数据库

典型产品：

- PostgreSQL
- MySQL

适合：

- 强一致业务数据
- 事务
- 复杂查询
- 结构化数据
- 报表和聚合

核心概念：

- 表
- 主键
- 外键
- 索引
- 事务
- 隔离级别
- 唯一约束
- 查询计划

事务 ACID：

- Atomicity：原子性
- Consistency：一致性
- Isolation：隔离性
- Durability：持久性

常见设计原则：

- 每张表有稳定主键
- 重要业务唯一性用唯一索引保证
- 状态字段使用明确枚举
- 时间字段至少包含 `created_at`、`updated_at`
- 软删除要明确查询过滤策略
- 大字段不要随意塞进主业务表

任务表例子：

```sql
CREATE TABLE agent_tasks (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  status TEXT NOT NULL,
  input JSONB NOT NULL,
  result JSONB,
  error_code TEXT,
  error_message TEXT,
  idempotency_key TEXT UNIQUE,
  created_at TIMESTAMPTZ NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL
);
```

状态机例子：

```txt
pending -> running -> succeeded
pending -> running -> failed
failed -> retrying -> running
running -> cancelled
```

不要允许任意状态互相跳转。

### 3.2 Redis

Redis 是内存数据结构服务，不只是缓存。

常见用途：

- 缓存
- 分布式锁
- 限流
- 排行榜
- 会话
- 短期状态
- Pub/Sub
- Stream 队列

常见数据结构：

- String
- Hash
- List
- Set
- Sorted Set
- Stream
- Bitmap

缓存模式：

Cache Aside：

1. 读缓存。
2. 缓存未命中读数据库。
3. 写入缓存。
4. 返回结果。

Write Through：

1. 写请求先写缓存层。
2. 缓存层同步写数据库。

Write Behind：

1. 先写缓存。
2. 异步写数据库。
3. 性能高但一致性风险更大。

常见缓存问题：

- 缓存穿透：查不存在的数据，每次都打到数据库
- 缓存击穿：热点 key 过期，大量请求同时打到数据库
- 缓存雪崩：大量 key 同时过期
- 缓存污染：缓存了错误或过期数据

解决方式：

- 空值缓存
- Bloom Filter
- 热点 key 永不过期或后台刷新
- 过期时间加随机抖动
- 单飞 singleflight，避免并发重复加载

### 3.3 向量库

向量库用于相似度检索，是 AI 记忆系统和 RAG 的基础组件。

常见产品：

- pgvector
- Milvus
- Qdrant
- Pinecone
- Weaviate
- Elasticsearch vector search

核心概念：

- embedding：文本、图片等内容的向量表示
- dimension：向量维度
- similarity：相似度，例如 cosine、dot product、L2
- index：向量索引，例如 HNSW、IVF
- metadata：过滤条件，例如用户、文档、时间、权限

向量库适合：

- 语义搜索
- 相似内容推荐
- 长期记忆召回
- 文档问答

向量库不适合单独承担：

- 权限真相
- 强事务数据
- 精确条件查询
- 复杂关系建模

AI 记忆系统常见结构：

```txt
PostgreSQL：记忆事实、权限、版本、来源
Object Storage：原始文档、图片、附件
Vector DB：语义向量和召回索引
Redis：短期会话、热点记忆、任务进度
```

记忆数据建议包含：

- `memory_id`
- `user_id`
- `scope`
- `content`
- `embedding`
- `source`
- `created_at`
- `updated_at`
- `expires_at`
- `confidence`
- `visibility`

常见坑：

- 只存向量，不存原文和来源
- 不做权限过滤，跨用户召回
- embedding 模型升级后新旧向量混用
- 召回结果不做重排和去重
- 把短期上下文和长期记忆混在一起

### 3.4 数据一致性

缓存和数据库之间要明确一致性策略。

强一致：

- 写入后立即读到最新值
- 成本高
- 适合余额、权限、订单状态

最终一致：

- 短时间内可能读到旧值
- 系统最终收敛
- 适合统计、推荐、搜索索引

常见策略：

- 写数据库后删除缓存
- 延迟双删
- 使用消息队列异步刷新索引
- 使用版本号避免旧数据覆盖新数据

实践建议：

- 权限、余额、订单不要只信缓存
- 搜索索引和向量索引允许最终一致
- 所有异步索引更新要能重放
- 数据修复脚本要有审计日志

## 4. 分布式系统

当系统从单体服务变成多个服务、多个队列、多个 worker、多 Agent 协作时，分布式系统问题会自然出现。

分布式系统的核心现实：

```txt
网络会失败
消息会重复
节点会宕机
时钟会漂移
状态会不一致
故障会传播
```

### 4.1 消息队列

消息队列用于解耦生产者和消费者。

常见产品：

- RabbitMQ
- Kafka
- Redis Streams
- AWS SQS
- Google Pub/Sub
- NATS

适合：

- 异步任务
- 削峰填谷
- 事件驱动
- 日志流水
- 多消费者协作

消息基本结构：

```json
{
  "event_id": "evt_01H...",
  "event_type": "task.created",
  "occurred_at": "2026-07-22T10:00:00Z",
  "payload": {
    "task_id": "task_123"
  }
}
```

关键概念：

- Producer：生产消息
- Consumer：消费消息
- Topic / Queue：消息通道
- Ack：确认处理完成
- Retry：失败重试
- DLQ：死信队列
- Offset：消费进度

消息语义：

- At most once：最多一次，可能丢
- At least once：至少一次，可能重复
- Exactly once：严格一次，成本高且通常有条件限制

大多数业务应该按“至少一次 + 幂等消费”设计。

### 4.2 幂等消费

消息队列可能重复投递，所以消费者必须幂等。

常见做法：

- 消息有全局唯一 `event_id`
- 消费前查去重表
- 业务写入和消费记录在同一事务
- 已处理消息直接 ack

处理流程：

```txt
收到消息
  -> 开事务
  -> 检查 event_id 是否已处理
  -> 执行业务写入
  -> 记录 event_id 已处理
  -> 提交事务
  -> ack 消息
```

### 4.3 一致性

分布式系统里，一致性需要明确选择。

常见模型：

- 强一致
- 最终一致
- 读己之写
- 单调读
- 因果一致

CAP：

- Consistency：一致性
- Availability：可用性
- Partition tolerance：分区容错

CAP 不是让你三选二背概念，而是提醒：网络分区发生时，系统必须在一致性和可用性之间做取舍。

业务选择：

- 支付、库存、权限：偏一致性
- 推荐、搜索、统计：偏可用性和最终一致
- Agent 任务状态：通常可最终一致，但状态机必须可恢复

### 4.4 分布式事务与 Saga

跨服务事务很难。常见替代方案是 Saga。

Saga 思路：

```txt
步骤 1 成功
步骤 2 成功
步骤 3 失败
  -> 执行步骤 2 的补偿
  -> 执行步骤 1 的补偿
```

例子：

```txt
创建订单
  -> 扣库存
  -> 扣余额
  -> 通知发货
```

如果扣余额失败，需要释放库存、取消订单。

Saga 要点：

- 每一步都要有明确状态
- 每一步都要幂等
- 补偿动作也要幂等
- 中间状态要可查询
- 失败后要能人工介入

### 4.5 容错

容错不是“不失败”，而是失败时系统仍可控。

常见手段：

- 超时
- 重试
- 限流
- 熔断
- 降级
- 隔离
- 快速失败
- 健康检查
- 自动恢复

熔断：

- 下游连续失败时，暂时停止调用
- 避免把请求继续打到故障服务
- 过一段时间半开试探恢复

降级：

- 推荐服务失败时返回热门内容
- 向量检索失败时退回关键词搜索
- 模型服务繁忙时返回排队状态

隔离：

- 不同租户隔离
- 不同任务队列隔离
- 高优先级任务和低优先级任务隔离
- 外部调用线程池隔离

### 4.6 多 Agent 协作

多 Agent 系统本质上是分布式任务系统。

常见组件：

- Orchestrator：编排器
- Planner：规划 Agent
- Worker Agent：执行 Agent
- Memory Service：记忆服务
- Tool Gateway：工具网关
- Queue：任务队列
- State Store：状态存储
- Observability：日志、追踪、指标

一个可靠的 Agent 任务应该有：

- `task_id`
- `parent_task_id`
- `agent_id`
- `status`
- `input`
- `output`
- `error`
- `attempt`
- `deadline`
- `idempotency_key`
- `trace_id`

多 Agent 常见问题：

- 循环调用
- 工具重复执行
- 中间状态丢失
- 子任务失败后主任务不知道
- 多个 Agent 同时写同一份记忆
- 没有全局 trace，无法复盘

工程建议：

- 所有 Agent 动作事件化
- 所有工具调用幂等化
- 任务状态持久化
- 子任务有超时和取消
- 共享记忆写入要有版本控制
- 编排器负责最终状态收敛

## 5. 可观测性

可观测性回答三个问题：

```txt
发生了什么？
为什么发生？
影响多大？
```

三大支柱：

- Logs：日志
- Traces：追踪
- Metrics：指标

### 5.1 日志

日志适合记录离散事件。

日志应该结构化：

```json
{
  "level": "error",
  "message": "task execution failed",
  "request_id": "req_123",
  "trace_id": "trace_abc",
  "task_id": "task_456",
  "agent_id": "agent_planner",
  "error_code": "MODEL_TIMEOUT",
  "duration_ms": 2100
}
```

日志级别：

- DEBUG：调试细节
- INFO：关键业务事件
- WARN：异常但可恢复
- ERROR：失败，需要关注
- FATAL：进程不可继续

日志实践：

- 使用结构化 JSON 日志
- 每条日志带 `request_id` 或 `trace_id`
- 不记录密码、Token、隐私数据
- 错误日志包含错误类型和堆栈
- 关键状态流转要有日志

常见坑：

- 日志只有一句“失败了”
- 没有 request_id，串不起调用链
- 打印完整用户输入，泄露隐私
- 成功路径日志太多，成本暴涨
- 错误被吞掉，没有任何日志

### 5.2 追踪

Trace 用于观察一次请求跨多个服务的完整链路。

核心概念：

- Trace：一次完整请求链路
- Span：链路中的一个步骤
- Trace ID：链路 ID
- Span ID：步骤 ID
- Parent Span：父步骤

例子：

```txt
HTTP POST /agent/tasks
  -> auth.check
  -> db.insert_task
  -> queue.publish
  -> worker.consume
  -> vector.search
  -> model.generate
  -> db.update_task
```

追踪能回答：

- 慢在哪里？
- 哪个下游失败？
- 重试了几次？
- 队列等待多久？
- Agent 哪一步产生了错误？

实践建议：

- 请求入口创建 trace
- 跨服务传递 trace context
- 数据库、Redis、HTTP、队列都打 span
- 长任务要把 trace_id 写入任务表
- Agent 工具调用要记录成 span

### 5.3 Metrics

Metrics 用于看趋势和告警。

四类黄金指标：

- Latency：延迟
- Traffic：流量
- Errors：错误率
- Saturation：饱和度

常见指标：

- QPS
- P50 / P90 / P95 / P99 延迟
- 错误率
- 超时率
- 重试次数
- 队列长度
- 消费延迟
- 数据库连接池使用率
- Redis 命中率
- CPU / 内存 / 磁盘
- 模型调用耗时和失败率

指标类型：

- Counter：只增计数，例如请求总数
- Gauge：可增可减，例如队列长度
- Histogram：分布，例如请求耗时

告警建议：

- 对用户影响告警，不要只对机器指标告警
- 告警要有行动手册
- 同一故障不要触发几十条重复告警
- 区分 warning 和 critical

### 5.4 AI / Agent 系统的可观测性

AI 应用除了传统指标，还要观察：

- Prompt 版本
- Model 名称
- Token 输入/输出
- 工具调用次数
- 工具失败率
- 检索命中文档
- 向量召回分数
- Agent 迭代轮数
- 任务取消率
- 人工接管率
- 输出质量反馈

Agent 任务日志建议：

```json
{
  "event": "tool.call",
  "trace_id": "trace_123",
  "task_id": "task_456",
  "agent_id": "researcher",
  "tool_name": "web_search",
  "input_hash": "sha256:...",
  "duration_ms": 823,
  "status": "succeeded"
}
```

不要把完整敏感输入直接打到日志里，可以存 hash、摘要或脱敏版本。

## 从前端到后端的学习路线

### 阶段 1：能稳定调用接口

目标：

- 理解 HTTP 语义
- 会设计错误码
- 会设置超时
- 会做安全重试
- 理解幂等

练习：

- 写一个任务创建 API
- 支持 `Idempotency-Key`
- 故意让服务随机超时
- 客户端做指数退避重试
- 验证不会重复创建任务

### 阶段 2：能处理并发

目标：

- 理解异步 I/O
- 会限制并发
- 会配置连接池
- 理解背压

练习：

- 写一个批量 URL 抓取服务
- 限制最大并发 10
- 下游慢时排队
- 队列超过上限时拒绝
- 记录每个任务状态

### 阶段 3：能管理状态

目标：

- 会设计关系表
- 会使用事务和唯一索引
- 会使用 Redis 缓存
- 理解缓存一致性
- 理解向量库用途

练习：

- 做一个文档记忆系统
- 文档元数据存 PostgreSQL
- 热点会话存在 Redis
- 文档片段写入向量库
- 查询时先权限过滤再召回

### 阶段 4：能做异步协作

目标：

- 理解消息队列
- 会做幂等消费
- 会设计任务状态机
- 会处理失败重试和死信

练习：

- 上传图片后异步生成缩略图
- API 只创建任务
- Worker 从队列消费
- 失败自动重试 3 次
- 最终失败进入死信队列

### 阶段 5：能定位线上问题

目标：

- 会打结构化日志
- 会串 trace
- 会定义 metrics
- 会写告警规则
- 会复盘事故

练习：

- 给任务系统加 request_id 和 trace_id
- 记录每个状态流转
- 暴露队列长度和任务耗时指标
- 模拟 worker 故障
- 根据日志和指标定位问题

## 后端工程检查清单

### 网络接口

- [ ] 每个外部调用都有超时
- [ ] 错误响应结构统一
- [ ] 有 request_id / trace_id
- [ ] 写操作考虑幂等
- [ ] 重试只针对可重试错误
- [ ] 鉴权和授权在后端执行

### 并发

- [ ] 并发数量有限制
- [ ] 连接池配置合理
- [ ] 队列有最大长度
- [ ] 慢任务不阻塞请求线程
- [ ] 有背压或降级策略
- [ ] 异步任务错误不会被吞掉

### 存储

- [ ] 关键业务唯一性由数据库约束保证
- [ ] 状态流转有明确状态机
- [ ] 缓存有过期和失效策略
- [ ] 缓存和数据库一致性策略明确
- [ ] 向量数据有来源和权限元数据
- [ ] 数据修复和重放有审计

### 分布式

- [ ] 消息有唯一 ID
- [ ] 消费者幂等
- [ ] 失败有重试和死信
- [ ] 任务状态可恢复
- [ ] 下游故障不会无限扩散
- [ ] 多 Agent 写共享状态有版本控制

### 可观测性

- [ ] 日志结构化
- [ ] 日志不泄露敏感信息
- [ ] Trace 能跨服务串起来
- [ ] Metrics 覆盖延迟、流量、错误、饱和度
- [ ] 告警对应用户影响
- [ ] 每个告警有处理手册

## 推荐项目：多 Agent 任务系统

如果要把上面五块串起来，可以做一个小型多 Agent 任务系统。

功能：

```txt
用户提交一个研究任务
  -> API 创建任务
  -> Planner 拆分子任务
  -> Researcher 检索资料
  -> Writer 生成报告
  -> Reviewer 检查结果
  -> 用户查询最终报告
```

需要用到：

- HTTP API：创建任务、查询任务
- 幂等：重复提交不重复创建
- 队列：异步执行子任务
- 数据库：任务、子任务、状态机
- Redis：任务进度缓存、限流
- 向量库：资料召回
- Trace：串起每个 Agent 调用
- Metrics：任务耗时、失败率、队列长度
- 日志：记录每个状态变更和工具调用

最小表设计：

```txt
tasks
subtasks
agent_runs
tool_calls
memories
processed_events
```

这类项目能很好地区分“会写 demo”和“懂后端工程”。

## 核心心智模型

后端工程不是把接口写出来，而是让系统在不完美环境下仍然可控：

```txt
请求会失败，所以要超时、重试、幂等。
并发会放大问题，所以要限流、连接池、背压。
状态会分裂，所以要事务、约束、状态机。
消息会重复，所以要幂等消费。
服务会宕机，所以要容错、恢复、补偿。
线上会出事，所以要日志、追踪、指标。
```

从“能调通”到“能稳定调通”，隔着的就是这些工程能力。
